# 博客邮箱订阅系统执行方案

## 目标

为当前 Hugo 博客增加一个自建的邮箱订阅系统，让访客可以输入邮箱订阅更新；当站点发布新博文后，系统自动向订阅用户发送邮件通知。

这个方案选择完全自建，目标不是最快上线，而是系统性锻炼后端能力，包括：

1. 表单处理
2. 数据存储
3. 邮件发送
4. 定时任务
5. 鉴权与安全
6. 退订与合规
7. 部署与运维

## 当前项目现状

当前博客是一个 Hugo 静态站点。

这意味着：

1. 前端页面可以继续由 Hugo 生成
2. 订阅能力不能只靠静态页面完成
3. 必须额外引入一个后端服务
4. 邮件发送、订阅数据存储、文章变更检测都需要后端负责

## 总体思路

整体拆成两个系统：

1. Hugo 博客前端
2. 自建订阅后端服务

推荐架构：

1. Hugo 负责展示订阅入口和成功/失败反馈页面
2. 后端负责处理邮箱提交、确认订阅、退订、检测新文章、发送邮件
3. 数据库存储订阅者、发送记录、文章发送状态
4. 定时任务定期检查博客 RSS 或内容变更

## 推荐技术路线

为了锻炼后端能力，同时保持复杂度可控，建议用这一套：

1. 后端语言：Node.js + TypeScript
2. Web 框架：Express 或 Fastify
3. 数据库：PostgreSQL
4. ORM：Prisma
5. 邮件服务：Resend、SendGrid 或 Mailgun
6. 定时任务：node-cron 或平台 Cron Job
7. 部署：Railway、Render、Fly.io 或 VPS

如果你更想练 Python，也可以替换为：

1. FastAPI
2. PostgreSQL
3. SQLAlchemy
4. APScheduler / Cron

但如果结合你当前网站结构和后续维护成本，我更推荐 Node.js + TypeScript。

## 核心业务流程

### 1. 用户订阅流程

1. 用户在博客页面输入邮箱并点击订阅
2. 前端将邮箱提交给后端接口
3. 后端校验邮箱格式
4. 后端检查该邮箱是否已存在
5. 如果不存在，则创建一条待确认订阅记录
6. 后端生成确认令牌
7. 后端发送确认邮件
8. 用户点击邮件中的确认链接
9. 后端验证令牌后，将订阅状态改为已确认

### 2. 新博文通知流程

1. 定时任务定期读取博客 RSS
2. 检测是否有未处理的新文章
3. 如果发现新文章，则读取文章标题、链接、摘要、发布时间
4. 查询所有已确认且未退订的订阅者
5. 生成邮件内容
6. 逐批发送通知邮件
7. 记录发送结果
8. 将该文章标记为已发送，避免重复发送

### 3. 退订流程

1. 每封邮件附带退订链接
2. 用户点击退订链接
3. 后端验证退订令牌
4. 将该邮箱标记为已退订
5. 返回退订成功页面

3. status

5. unsubscribe_token

7. unsubscribed_at

字段建议：

1. id

4. status
5. provider_message_id
6. error_message
7. created_at

作用：

1. 记录每封邮件的发送状态
2. 方便调试失败原因

## API 设计

### POST /api/subscribe

用途：提交邮箱订阅

请求体：

```json
{
  "email": "user@example.com"
}
```

返回结果：

1. 提交成功，提示用户去邮箱确认
2. 已订阅，给出友好提示
3. 参数错误，提示邮箱格式不合法

### GET /api/confirm

用途：确认订阅

查询参数：

1. token

结果：

1. 确认成功页面
2. 令牌失效页面

### GET /api/unsubscribe

用途：退订

查询参数：

1. token

结果：

1. 退订成功页面
2. 令牌失效页面

### POST /api/jobs/check-posts

用途：手动触发文章检查任务

说明：

1. 这个接口不对外公开
2. 只允许内部调用或管理员调用
3. 本地调试时很有用

## 邮件设计

需要准备三类邮件模板：

1. 订阅确认邮件
2. 新文章通知邮件
3. 退订确认页文案

### 订阅确认邮件

内容建议：

1. 感谢订阅
2. 提示点击按钮确认
3. 如果不是本人操作可忽略

### 新文章通知邮件

内容建议：

1. 文章标题
2. 发布时间
3. 文章摘要
4. 阅读原文按钮
5. 退订链接

## 新文章检测策略

推荐第一版直接基于 RSS 检测，因为最简单，也最适合 Hugo。

做法：

1. 定时读取站点 RSS 地址
2. 解析最近若干篇文章
3. 用文章唯一标识判断是否已发送

可用唯一标识：

1. guid
2. link
3. 标题 + 发布时间的组合

优先推荐：

1. guid
2. 如果 RSS 中没有稳定 guid，则使用 link

## 安全与合规要求

这一部分是方案 C 必须认真做的，否则系统上线后风险很高。

### 必做项

1. 所有表单做服务端校验
2. 限流，避免被恶意刷接口
3. 邮箱唯一索引，避免重复数据
4. 订阅必须双重确认
5. 每封邮件必须包含退订链接
6. 退订后不得继续发送
7. 敏感配置放到环境变量

### 建议做项

1. 增加验证码或隐藏蜜罐字段防机器人
2. 记录 IP 和 User-Agent 用于风控
3. 订阅接口加频率限制
4. 邮件模板中注明订阅来源

## 前端接入方案

虽然这一版重点是后端，但前端最终仍需要一个入口。

建议第一版把订阅入口放在：

1. 侧边栏
2. 首页底部
3. 每篇文章底部

最简形式：

1. 一个邮箱输入框
2. 一个订阅按钮
3. 一个简短说明文字

示例文案：

1. 输入邮箱，博文更新后自动通知我
2. 需要一次邮件确认，可随时退订

## 风险与难点

### 难点一：邮件送达率

问题：

1. 邮件可能进垃圾箱
2. 域名信誉影响送达率

解决方向：

1. 使用专业邮件服务
2. 正确配置发信域名
3. 使用双重确认减少垃圾投诉

### 难点二：重复发送

问题：

1. 定时任务多次执行时可能重复通知

解决方向：

1. 建立文章发送记录表
2. 增加唯一约束
3. 使用幂等逻辑

### 难点三：接口被刷

问题：

1. 订阅接口可能被机器人攻击

解决方向：

1. 加限流
2. 加验证码或蜜罐字段
3. 记录异常行为

## 最终建议

这个方案很适合练后端，但不要一口气把所有功能同时做完。

推荐实施顺序：

1. 先把订阅与确认流程做通
2. 再接 RSS 检测和自动发送
3. 最后补退订、限流、上线细节

这样能在最短时间内形成一个可运行的最小版本，同时又保留足够多的后端训练价值。
